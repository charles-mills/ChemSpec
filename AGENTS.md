# ChemSpec agent guide

This file applies to the whole repository. Keep changes aligned with the
project's central trust boundary:

> The agent produces text; the validator produces trust; the chemistry engine
> produces meaning; the application produces the experience.

No layer may bypass the layer immediately downstream of it. In particular,
unvalidated or stale source must never reach the simulation.

## Start with the relevant source of truth

Read only the documents needed for the task, but read the governing document
before changing a contract:

- `README.md` — product summary, current implementation status, and common
  commands.
- `docs/product-spec.md` — audience, supported chemistry, canonical journey,
  and product states.
- `docs/system-architecture.md` — crate ownership, dependency direction,
  shared contracts, Iced model, and GPU boundary.
- `docs/verification.md` — required invariants and test strategy by subsystem.
- `docs/safety.md` — virtual-only boundary, request dispositions, privacy, and
  adversarial cases.
- `docs/agent-workflow.md` — provider-neutral orchestration, evidence,
  structured results, and bounded repair.
- `docs/chems-specification.md` — normative `.chems` language semantics.
- `grammar/chems.ebnf` — normative grammar.
- `docs/chems-implementation-plan.md` — language implementation slices and
  acceptance criteria.
- `conformance/README.md`, `conformance/requirements.json`, and
  `conformance/manifest.json` — executable language and domain contract.
- `docs/implementation-plan.md` and `docs/delivery-plan.md` — task IDs,
  dependencies, integration gates, and delivery cut line.

Do not infer language semantics from the README or the example fixture when the
specification, grammar, or conformance registry answers the question. Do not
silently change a shared contract without updating its producers, consumers,
fixtures, and governing documentation in the same change.

For external libraries, frameworks, SDKs, APIs, and CLI tools, consult current,
version-matched documentation before coding. Use Context7 when available and
prefer official sources. This workspace pins Iced exactly to `0.14.0` and Rust
to `1.96.1`; do not write against a newer API by accident.

## Repository boundaries

- `crates/chem-domain`: exact, deterministic domain primitives. No parsing,
  networking, Iced, or GPU dependencies.
- `crates/chems-lang`: lossless source frontend, diagnostics, and formatting.
  It parses syntax; it does not decide whether chemistry is supported.
- `crates/chems-conformance`: validates the repository's executable
  specification and reports coverage.
- `crates/chemspec-app`: Iced composition and presentation. It may display
  validation but may not confer it.
- `fixtures`: reviewed integration artifacts. Expected chemistry must be
  independently authored or reviewed, never generated by the implementation
  under test and accepted as its own oracle.

Planned crates should preserve the boundaries in `docs/system-architecture.md`:
only the chemistry engine constructs `ValidatedExperiment`; the simulation
consumes validated meaning and remains renderer-independent; provider code
returns source and provenance, never trusted chemistry.

Keep dependencies pointing toward stable domain types. Do not introduce Iced
types into domain, language, validator, provider, or simulation contracts.

## Working rules

- Establish the relevant task ID, acceptance criteria, and integration gate
  from `docs/implementation-plan.md` before implementing planned work.
- Prefer the smallest complete vertical change over speculative abstractions or
  chemistry breadth outside the current closed-world scope.
- Preserve the distinction between `Validated`,
  `ValidatedWithAssumptions`, `Unsupported`, and `Invalid`.
- Preserve exact quantities. Chemistry and serialization code must not replace
  the repository's rational/decimal model with binary floating point.
- Keep output deterministic: serialization, diagnostics, formatting,
  catalogue digests, fixtures, and simulation seeds must be reproducible.
- Treat diagnostic codes, severities, and byte spans as compatibility
  surfaces. Update their conformance cases deliberately.
- Keep source provenance separate from `.chems` source and map evidence to
  claims, not merely to a general bibliography.
- Never expose hidden model reasoning. UI workflow entries are concise product
  events, evidence, and diagnostics.
- Never read credential files, log secrets, place them in prompts, or weaken a
  sandbox. Normal tests must use fakes and consume no subscription or API
  usage.
- Do not add actionable real-world hazardous procedures. Apply
  `docs/safety.md` whenever a change touches prompts, research, catalogue facts,
  generated explanations, or user-visible chemistry.
- Keep platform-specific code at narrow adapter boundaries.
- Preserve unrelated work in a dirty worktree and avoid broad formatting or
  cleanup unrelated to the task.

## Rust conventions

- The workspace uses Rust 2024, forbids `unsafe`, and enables Clippy `all` and
  `pedantic`. New crates should inherit workspace package metadata and lints.
- Model invariants with types and restricted constructors. In particular,
  trusted validated values must not be constructible through ordinary public
  fields.
- Prefer exhaustive typed enums for states, messages, diagnostics, and failure
  modes over booleans or stringly typed control flow.
- Propagate errors with context at subsystem boundaries. Do not convert provider
  failures, parse errors, unsupported chemistry, or validation failures into a
  successful result.
- Keep pure domain and transition logic separate from filesystem, process,
  network, clock, and renderer effects so it can be tested without those
  services.
- Avoid nondeterministic iteration in canonical output. Sort explicitly when
  map or set order could affect serialized bytes, diagnostics, or digests.

## Iced 0.14 and the Elm Architecture

Follow Iced's State -> Message -> Update -> View data flow in
`crates/chemspec-app`.

### State

- Application state is the single source of truth for product and UI state.
  Represent meaningful modes with enums instead of clusters of flags.
- Keep authoritative chemistry outside UI state. The app stores or references
  parser/validator/simulation outputs; it does not recreate their conclusions.
- Give every asynchronous request, validation, repair, or simulation generation
  a monotonically changing ID. Ignore completion messages whose ID is no longer
  current.
- Store durable state, not derived widget trees. Compute straightforward
  presentation values in `view`.

### Messages

- `Message` values describe user intent or completed external events, not
  imperative UI operations. Prefer names such as `RequestSubmitted`,
  `ValidationFinished`, and `PlaybackPaused`.
- Keep payloads typed, including request IDs and `Result` values. Derive
  `Debug` and `Clone` where Iced requires them.
- Feature modules may define their own message enum. The parent should route or
  map child messages explicitly so cross-feature effects remain visible.

### Update

- `update` is the only place that mutates application state. Keep each branch
  small, deterministic, and directly testable.
- A purely synchronous update may return `()`. Return `Task<Message>` once the
  transition launches an effect; use `Task::none()` when a branch has none.
- Start asynchronous work with Iced `Task`s and feed completion back through a
  typed message. Do not mutate application state from a background future or
  create an untracked runtime/event loop.
- Use `Task::batch` only for independent effects. Make ordering explicit when
  one effect depends on another.
- Invalidate a previously validated result immediately when its source or
  governing catalogue changes. Playback stays blocked until the current
  generation validates again.

### View

- `view(&self)` renders current state and wires widgets to messages. It must not
  perform I/O, block, mutate state, validate chemistry, or launch work.
- Decompose by product feature when a screen becomes difficult to read. A
  feature can expose focused `view`, `update`, and subscription functions
  without inventing an object-oriented widget hierarchy.
- Keep accessibility, empty/loading/error/unsupported states, and the persistent
  simulation disclosure visible. Never present a placeholder or provider error
  as a validated chemistry result.
- Use built-in Iced widgets, Canvas, and mesh primitives before custom renderer
  machinery. Do not create a second window surface, GPU adapter, device, or
  event loop.

### Tasks and subscriptions

- Tasks represent finite effects such as loading, parsing, validation, provider
  calls, saves, and focus operations. Their outputs return to `update` as
  messages.
- Subscriptions represent continuing external event streams. Derive them from
  current state and return `Subscription::none()` when the provider stream,
  timer, or playback is inactive.
- Avoid unconditional tick subscriptions. Subscribe only while animation or
  playback needs ticks, and make speed alter presentation rather than chemical
  meaning.
- Cancellation and timeout are product states. Stop the underlying work where
  supported and still reject late messages using generation IDs.

### Canvas and simulation presentation

- The simulation model owns identities, stages, ratios, and reaction extent;
  Canvas owns drawing and interaction only.
- Drawing must be deterministic for the same `SimulationFrame`. Illustrative
  randomness must use an explicit stable seed.
- Clear `canvas::Cache` whenever geometry-affecting state changes. Do not cache
  meaningful simulation state inside the renderer.
- Keep meaningful simulation behavior testable without a GPU. Visual tests are
  reserved for properties that cannot be expressed as state invariants.

## macOS Computer Use

For automated visual checks on macOS, never target the release-named `ChemSpec`
app and do not rely on a raw `cargo run` process. Computer Use may resolve that
name to an older registered bundle.

Run `just agent-smoke builder`, `just agent-smoke 2d`, or `just agent-smoke 3d`,
then target the app named `ChemSpec Agent Smoke`. Before judging a screenshot,
verify that its window title is respectively `ChemSpec Agent Smoke — Builder`,
`ChemSpec Agent Smoke — Structural 2D`, or
`ChemSpec Agent Smoke — Structural 3D`. The recipe rebuilds and byte-checks the
dedicated bundle before launching it. Run `just agent-smoke stop` when the live
check is complete.

## Verification

Run the narrowest relevant tests while iterating, then finish with the checks
that match CI:

```sh
cargo fmt --all --check
cargo test --workspace --all-targets
cargo clippy --workspace --all-targets -- -D warnings
cargo run -p chems-conformance -- validate
```

Useful focused commands include:

```sh
cargo test -p chem-domain
cargo test -p chems-lang
cargo test -p chems-conformance
cargo check -p chemspec-app
cargo run -p chemspec-app
cargo run -p chems-conformance -- report
```

`chems-conformance report` is a coverage report and intentionally exits with
status 3 while requirements remain uncovered; do not describe that expected
status as a green release gate. GUI startup, live providers, platform
packaging, credential storage, and GPU initialization require explicit smoke
tests and must not be claimed from unit tests alone.

When handing work off, state the exact commands run, what passed, and any
boundary that still needs live, GPU, network, credential, or cross-platform
verification.
