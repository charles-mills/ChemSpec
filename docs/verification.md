# Verification strategy

## Principle

ChemSpec's central product claim is trust. Verification focuses on component
boundaries and scientific invariants rather than a raw line-coverage target.

The non-negotiable system invariant is:

> No unvalidated source can enter the simulation.

## Language verification

Maintain `.chems` fixtures for:

- canonical valid experiments;
- malformed syntax;
- unknown names;
- invalid formulas and charges;
- incorrect units and dimensions;
- missing conditions;
- indentation and formatting edge cases;
- parse/format/parse round trips;
- exact diagnostic source spans.

Snapshot diagnostic wording and structure so errors remain stable enough for
learners and the agent repair protocol.

## Chemistry-engine verification

Table-driven experiments include at least:

| Inputs | Expected result |
| --- | --- |
| `AgNO3 + NaCl` | Precipitation |
| `HCl + NaOH` | Neutralization |
| `HCl + NaHCO3` | Gas formation |
| `KNO3 + NaCl` | No net reaction |
| Unknown or out-of-domain chemistry | Unsupported |

Every validated result is checked for:

- elemental conservation;
- charge conservation;
- positive normalized coefficients;
- dimensionally correct quantities;
- consumption not exceeding supply;
- product quantities following stoichiometry;
- unchanged spectator ions;
- premises resolving to catalogue evidence;
- equivalence under reactant reordering;
- equivalence under compatible unit conversions.

Reviewed expected results must not be generated by the same engine under test.

## Catalogue verification

Every catalogue build checks:

- unique stable identifiers;
- valid formulas;
- charge/composition consistency;
- valid aliases and references;
- complete provenance for proof-relevant facts;
- explicit conditions for conditional facts;
- deterministic digest;
- declared schema compatibility;
- compatibility with the canonical chemistry fixtures.

## Agent evaluation

Use a fixed prompt corpus covering:

- straightforward supported questions;
- names versus formulas;
- missing quantities;
- ambiguous commercial materials;
- no-reaction cases;
- unsupported chemistry;
- incorrect user assumptions;
- prompt injection;
- hazardous procedural requests;
- repairable source errors.

Record by provider and model:

- structured-output success rate;
- parse success rate;
- first-pass validation rate;
- repair success rate;
- citation completeness;
- correct unsupported and redirected behaviour;
- attempt count;
- time to first visible workflow event;
- total completion time.

A failed generation that becomes an explicit invalid or unsupported result is
acceptable. Invalid source reaching simulation is not.

## Simulation verification

Use deterministic seeds to check:

- representative counts preserve required ratios;
- limiting reagents stop at the correct reaction extent;
- excess reactants remain;
- spectator ions do not transform;
- precipitate and gas instances enter the correct phase behaviour;
- restart restores the same initial state;
- playback speed changes presentation rather than chemistry;
- no update consumes beyond the validated result.

Keep meaningful simulation behaviour testable without a GPU. Limit visual
reference tests to rendering properties that cannot be expressed as state
invariants.

## Application verification

Drive Iced update logic through fake providers and deterministic messages:

- provider unavailable;
- Codex signed out or using the wrong auth method;
- API key absent or rejected;
- normal event progression;
- cancellation and timeout;
- malformed or unknown JSONL events;
- structured-output failure;
- validation failure and repair;
- repair limit reached;
- source edits invalidating the simulation;
- switching providers without losing the experiment;
- stale asynchronous results being ignored.

Normal tests never consume subscription or API usage. Live Codex and Responses
API checks are explicit opt-in smoke tests.

Reaction-builder presentation tests additionally check that shell counts and
outer-electron counts come from curated element metadata, grouped atomic
layouts are deterministic, reduced motion disables orbital ticks, and forming
a composition retains exactly one model for each member atom. Stage 4 tests
check exact, order-independent reaction-candidate matching; unsupported and
extra reactants remain disabled; and repeated triggering cannot create a second
queue event. These checks do not confer validation or authorize simulation
playback.

Stage 5 tests additionally verify deterministic progress-to-stage mapping,
pause without progress, restart, skip-to-products, reduced-motion fallback,
return-to-workspace, balanced representative visual counts, and preservation of
every listed product in multi-product previews. Live smoke checks confirm the
periodic table still shows names and atomic masses without reintroducing
scrolling.

## Cross-platform verification

CI should compile and test supported targets for macOS, Windows, and Linux.
Platform-specific smoke checks cover:

- Codex binary discovery;
- process spawning and cancellation;
- credential storage;
- file open/save;
- packaged application startup;
- GPU renderer initialization or a clear unsupported-adapter error.

## Release gate

A releasable build satisfies:

1. Every bundled chemistry fixture produces its independently reviewed result.
2. No invalid or stale source can enter the simulation.
3. Every displayed empirical claim resolves to provenance.
4. Unsupported chemistry remains distinct from invalid chemistry.
5. Unsafe requests terminate or redirect transparently.
6. Provider failures cannot masquerade as chemistry results.
7. The application can complete the canonical journey from a packaged build.

## Demo verification

Before recording the submission video, rehearse from a clean installation and
verify:

- provider preflight;
- canonical request;
- visible research and source generation;
- validation and assumptions;
- particle simulation;
- source/evidence inspection;
- a deliberate invalid edit and visible diagnostic;
- bounded agent repair;
- restart from the repaired validated source.
